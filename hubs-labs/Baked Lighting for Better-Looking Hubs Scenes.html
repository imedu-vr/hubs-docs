Baked Lighting for Better-Looking Hubs Scenes

May 24, 2022 ‚Äî 28 min read

Light baking is a hot topic in the world of virtual environments. This article will reveal why it's so important for making Hubs scenes look and perform their very best on all devices.

Many creators that are relatively new to making content for virtual platforms have struggled with getting things to run smoothly, especially in VR headsets. Most of the challenges have to do with hardware limitations and how much effort it takes for devices to display so much with such relatively underpowered chipsets. While this will inevitably improve as time goes on, there are some clear steps you can take today that will make your scenes run more smoothly on all devices.

In this tutorial, we‚Äôll be looking to eliminate the biggest enemy of smooth real-time graphics: dynamic lights. <cue crash of thunder and lightning> (Actually, don't. That would be another dynamic light.)

If you're viewing this tutorial on a phone, you'll have a better experience holding it in landscape (horizontal) orientation.

What is a dynamic light?

Any light that is placed in a scene and generates light onto the environment and avatars is considered to be dynamic in that, in theory, you could move it around, change its color or brightness, or even delete it and you‚Äôd see the scene change in response.

We can illustrate this by attaching a light to an animated object or even an avatar. The results can be quite compelling, but don‚Äôt get too excited‚Äìeven one dynamic light can make your scene too complex to run on certain devices.

You might be thinking, ‚ÄúBut none of my lights are moving around? Shouldn‚Äôt that be ok?‚Äù. The fact is, those lights are still being calculated in real time despite them sitting in one location. Therefore, we are still incurring a rendering cost and it gets worse with each light that‚Äôs added to the scene. Sure, it may run just fine on a desktop computer with a fancy graphics card. But it‚Äôs important to remember that people join Hubs rooms from all over the world using all kinds of devices, including mobile phones. Ensuring everyone has access to virtual spaces should be important to anyone who cares about maximizing the reach of their work.

So what's the alternative to dynamic lights?

The good news is that we can achieve similar results with baked lighting combined with reflection probes and we won‚Äôt need any dynamic lights whatsoever. The extra good news is that this means we can make scenes that actually look even better than their dynamically-lit versions since we can harness the power of bounced lighting. And then the extra extra (I promise I‚Äôll stop) good news is that these even-better looking scenes can run on mobile phones and VR headsets.

What's the bad news then?

Well, like most things that look good, it‚Äôs going to require some work. But I think you‚Äôll find the end result is so good that you‚Äôll want to redo all your scenes.

Lighting the Scene

We Need a 2nd UV Layer

This is often the most misunderstood part of light baking so don‚Äôt be surprised if you need to go through this section a few times. It‚Äôs actually more difficult to explain so I would first highly recommend reading the beginning of this other Creator Labs article to get a grasp on what UVs are.

If we wanted to store the light and shadow information and put it onto the surfaces of the scene, how would we know where to put them? This is where a 2nd layer of UVs comes into play. We need to unwrap all the surfaces of all the objects and make sure that none of the faces overlap. This is quite different from how we might unwrap them for regular texturing (like adding bricks to the wall, for example). In regular texture unwrapping, it‚Äôs often desirable to have parts overlapping so that you can reuse parts of the texture or have repeating textures that tile horizontally or vertically. But with light mapping, every single surface is getting unique lighting so they need their own unique spot on the UV layout. Go ahead and read this whole paragraph again because it‚Äôs very important.

Each object in the scene will require a new UV layer that is set up specifically for light baking.

There are many ways to accomplish this. I‚Äôm going to show you how I like to do it, but bear in mind that every situation and project is different so you may choose to experiment once you know how it all works.

The basic steps are:

  1. Duplicate the existing UV layer
  2. Re-unwrap the object

Step 1 looks like this:

It‚Äôs important from this point forward that we keep that new UV layer selected on all the objects. If you‚Äôre not sure what I mean, I‚Äôm talking about keeping that newly created layer called ‚ÄòUVMap.001‚Äô highlighted:

When we do our light baking, this is how Blender knows which layer to operate on. We don‚Äôt want to accidentally render over any textures our objects may already have on the first UV layer.

Unwrap the 2nd UV Layer

Step 2 is the unwrapping part. One cool thing about Blender is that it allows you to unwrap all your objects at once. The benefit of this method is that you can pack all those UV islands into one single texture if you wish. The other benefit is that a single lightmap texture can be faster to download (remember, Hubs runs in a browser). Also, if your objects all share the same material(s) already, having 1 lightmap texture is less work on your graphics card when running Hubs.

I like to do a sort of hybrid approach where I unwrap each object separately, but then pack them all together at the end. The benefit of unwrapping each object individually is that it‚Äôs often easier to understand what you‚Äôre looking at and you can optimize each object‚Äôs UVs without the clutter of all the other objects confusing things.

The only option that I sometimes adjust is the ‚ÄòAngle Limit‚Äô. Lower numbers cause the unwrap to make more islands, whereas higher numbers produce fewer islands but with less distortion. Having faces take up more UV space means that the resolution of the texture will be higher on those areas. It also means you can make room for more objects.

Don‚Äôt be surprised if you find yourself spending lots of time iterating on this step. A well-packed UV layout is a thing of beauty and it‚Äôs not something you want to have to redo once you‚Äôve moved on to other tasks.

Material Setup

This is the last big step before we actually perform the light bake, but it‚Äôs a very important one. We need to tell Blender exactly what and where we‚Äôd like the lighting to be baked. This involves going into each material and setting up a target texture. I‚Äôll set it up on one material, then copy it to the others.

In Blender‚Äôs Shader Editor, we need to add an Image Texture node and create a new blank image while leaving it unhooked from the graph. The setup of that image is crucial. The size of the image will determine how detailed the light and shadows appear, but bear in mind that you can often get away with lower resolution lightmaps than you might think. I like to use a fairly large texture (2048 or even 4096) because I can easily scale it down later.

The other crucial step is checking that ‚Äò32-bit‚Äô checkbox. We need this lightmap texture to have a high dynamic range. In other words, we need the pixel values (brightness) to be able to go above and beyond what they normally can reach (0-255). That‚Äôs essentially what will make the lighting look realistic, especially when you have bright sunlight.

Bake It Till You Make It

This is the part where we need to make sure we‚Äôve got everything ready. I go through a mental checklist:

  • Are all my lights set up the way I want?
  • Are there any hidden objects that might interfere with my lighting?
  • Do all my objects have a 2nd UV channel?
  • Are all my objects unwrapped and have non-overlapping UVs?
  • Are all the objects I‚Äôm baking selected?
  • Do my selected objects have a target texture for baking?
  • Is the target texture node selected in all the materials I‚Äôm baking?

If you answered ‚ÄòYes‚Äô to all of these questions, you‚Äôre ready to bake.

So now we venture into the Render Properties tab which is where baking is started.We need to make sure we‚Äôre using the ‚ÄòCycles‚Äô renderer, otherwise we won‚Äôt get all that nice bounced lighting and everything that makes the lighting look so good.

Now, the settings for rendering can be daunting to look at, but we don‚Äôt need to worry about most of the settings as the defaults are generally pretty close to what we‚Äôll be using. The only caveat here is that it‚Äôs sometimes helpful to bake with lower quality at first while testing. Then, once you know things are working, you can turn up the quality level for a final bake.

Under the ‚ÄòSampling‚Äô-->‚ÄôRender‚Äô section, I usually set the ‚ÄòMax Samples‚Äô to a lower number like 1024. I leave the rest of the values alone for now.

üí°
Make sure you‚Äôre adjusting the ‚ÄòRender‚Äô samples. It‚Äôs easy to confuse them with the ‚ÄòViewport‚Äô samples right above them. Those ‚ÄòViewport‚Äô settings only affect the 3d viewport inside Blender and have no effect on your bake.

Still in the ‚ÄòRender Properties‚Äô tab, now we can scroll down to the ‚ÄòBake‚Äô section. From here, we change the ‚ÄòBake Type‚Äô to ‚ÄòDiffuse‚Äô. We don‚Äôt actually want the color of the surface textures to get baked into our lighting, so we uncheck ‚ÄòColor‚Äô but leave ‚ÄòDirect‚Äô and ‚ÄòIndirect‚Äô lighting contributions on.

Second, under ‚ÄòMargin‚Äô, we adjust the ‚ÄòSize‚Äô value to be a little lower. This parameter adds extra pixels around the perimeter of each UV island. That way, if an island is right on the edge of a rendered pixel and the background, those extra pixels will keep the model from having visible seams. You can always adjust this value after baking if it‚Äôs too high or too low. My UV islands are pretty tightly packed so I lower it quite a bit.

Last, we press the big ‚ÄòBake‚Äô button and wait‚Ä¶

The status bar at the very bottom of Blender will display the progress. Depending on the complexity of the scene, lighting, texture resolution, and render samples, this baking process can take a long time. Don‚Äôt be surprised if it takes many minutes‚Äìeven hours.

For this particular scene, it‚Äôs a few minutes. If I first switch the renderer to use my GPU, it drops to mere seconds.

As soon as it‚Äôs finished baking, that status bar displays a message that reads:

In the Save dialog that appears, we‚Äôll save our image as a ‚ÄòRadiance .HDR‚Äô file.

If your baked lightmap image turns out kind of grainy or noisy, you can turn up the Renderer‚Äôs ‚ÄòMax Samples‚Äô value a bit and re-bake it. I will probably raise mine to 4096. The other helpful option you can try is that ‚ÄòDenoise‚Äô checkbox. It uses the ‚ÄòNoise Threshold‚Äô value above to determine how much to reduce that graininess. üí°

Preparing for Export

We‚Äôve made it to the home stretch (of lighting anyway)! The last step is to get these lightmaps exported properly with our scene. For this, we‚Äôll need to make sure we‚Äôre using the Hubs Blender Add-on.

I‚Äôve provided a link above in case you need to install it. There‚Äôs only one thing we need from it and it‚Äôs available inside the Shader Editor.

For each material that has baked lighting, we need to add the node called ‚ÄòMOZ_lightmap‚Äô. Our Image Texture node with the lightmap on it will have it‚Äôs output connected to this new node.

We must also tell our ‚ÄòImage Texture‚Äô node which UV layer to use. For this, we add a ‚ÄòUV Map‚Äô node and pick our ‚ÄòUVMap.001‚Äô from its dropdown menu. The output of the ‚ÄòUV Map‚Äô node will get connected to the ‚ÄòVector‚Äô input of the ‚ÄòImage Texture‚Äô node.

Once this has been set up for one material, we can simply copy those three connected nodes to all the other materials.

To see these lightmaps in all their glory, we need to export the scene. Remember, there‚Äôs no need to export any lights since they are only used in Blender and we‚Äôve already gotten what we need from them. I like to keep my lights in their own Collection in case I need to re-bake things at some point.

I‚Äôm exporting the whole scene as one big .glb file. Don‚Äôt be shocked if the file is larger than you expected. Those .HDR files can get pretty huge, which is why I often reduce their resolution. Setting up the Background Environment

One thing that may not be immediately obvious is that Blender‚Äôs background has an effect on the lighting. If you go to the ‚ÄòWorld Properties‚Äô tab, you‚Äôll see there‚Äôs a default background color that is used in Blender‚Äôs 3d viewports. By changing that color swatch, you can actually see it affect the rendering of the scene. Guess what? It affects the light bake too.

I haven‚Äôt given much thought to what the background should look like in Hubs, but even a light blue sky would be better than the default black background I have now. But there‚Äôs yet another way to use this background to our advantage. We can use a 360-degree panoramic HDR image as a background and it will emit light in all directions based on its values and hues. This may be done instead of using Blender lights OR in conjunction with them. Just be aware that the look of the HDR image could interfere with your Blender lighting. For example, if the image has a bright sun in it, that may give the appearance of an extra sun since it gets doubled with your Blender light. Choosing a good HDR image can be tricky, but you can find lots of them for free at Polyhaven.com where people have kindly donated many different assets with Creative Commons licenses.

Reflections

The scene is really looking good now, but there‚Äôs still something missing. You may not have realized it if you visited your room alone. But if you press i in Hubs, or if you drag/drop a 3d model into your browser window, you‚Äôll soon see a problem. None of the avatars or objects have any lighting on them!

To remedy this situation, I‚Äôm heading back into Blender. We‚Äôll be adding something called a ‚ÄòReflection Cubemap‚Äô to our scene. Using the Hubs Blender Add-on tooling, we can render our own 360-degree HDR image that may be used as a background image and/or used as something to create reflections.

From the ‚ÄòAdd‚Äô menu, we‚Äôll choose ‚ÄòLighting‚Äô-->‚ÄôReflection Cubemap‚Äô. Then we‚Äôll move it around so the center is sitting roughly at eye-level in the center of our scene. We‚Äôll be adjusting its ‚ÄòSize‚Äô parameter to encompass the geometry, but you can also use the scale tool for non-uniform shapes. The size of the reflection cubemap is important because it needs to cover any geometry you want affected by its reflection.

We‚Äôre going to use this reflection cubemap as a quick way to render a 360-degree panorama from that spot. The resulting image can be used to reflect on the surfaces of anything inside that cube. What‚Äôs extra cool is that we‚Äôll see subtle reflections on background models too, adding an extra layer of realism to our lighting. The shinier (less rough) a model is, the clearer the reflection will appear. A fully metallic, zero-roughness object will look like a reflective chrome material.

We need to go to the reflection cubemap‚Äôs Object Properties tab and add the Hubs component called ‚ÄòReflection Probe‚Äô. This will show a dropdown menu where you may set the resolution of the reflection image it will create. The default works fine. You might be tempted to set this very large, but trust me, unless your scene is full of mirror-like surfaces, stick with a low resolution and it will still look good while not requiring users to load a giant texture they only see in reflections.

The Hubs team made this component for quick reflection textures, but it‚Äôs a useful tool for any Blender project where you need to render a 360-degree panorama. Sure, you can do it with Blender‚Äôs 360-equirectangular camera type, but this component eliminates a few steps.

When you press the big ‚ÄòBake‚Äô button and wait a moment, it generates the 360-degree equirectangular image and automatically saves it in a subfolder called ‚Äògenerated_cubemaps‚Äô.

It‚Äôs a bit of a ‚Äòset it and forget it‚Äô tool, but you must remember to re-bake any and all reflection cubemaps if your scene changes in any meaningful way.

Opening an ‚ÄòImage Editor‚Äô window and using the dropdown menu, you can find the image called something like ‚Äògenerated_cubemap-ReflectionCubemap‚Äô. (The name matches the object name.)
