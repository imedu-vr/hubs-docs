Polygons vs. Textures (or 'the starry sky tutorial')

Aug 22, 2022 ‚Äî 14 min read

The advice to use textures instead of polygons is quite common. In this tutorial, we'll be using some advanced modeling techniques that might make you rethink that advice.

Since the inception of 3d computer graphics, people have looked for ways to reduce complexity so that hardware can perform better. This goes for rendering as well as real time graphics‚Äìgetting a computer to draw stuff on screen can be surprisingly complex. When looking at a particular 3d object or rendering, a common refrain among so-called experts goes something like this: ‚ÄúHey, you know, you could‚Äôve made that with a texture instead of all those triangles.‚Äù
This know-it-all sentiment, while usually coming from a well-meaning place, is not always the best advice. In fact, sometimes it‚Äôs the exact opposite.

First, let‚Äôs show an example of when the old ‚Äútexture-is-cheaper-than-polygons‚Äù advice makes sense. Fortunately, I‚Äôve already written a tutorial that covers this concept. The ‚Äòtexture-is-cheaper-than-polygons‚Äô argument, while often true, is definitely not universal. Instead, it‚Äôs a more of a helpful reminder that there may be an alternative to how you might create something. In cases like the chain in the article linked previously, it‚Äôs pretty obvious that a 2-triangle object with a texture will be less taxing than an object with tens of thousands of triangles. But not every situation is as clear-cut.

Oh, my stars!

I recently came across a challenge that made me rethink my approach to modeling and texturing. I was deciding what kind of sky I wanted in a particular scene and settled on the idea of a late-evening sky where you could start to see stars appearing in the darker areas and less so near the sunlit horizon.

The thing I soon discovered was that stars in a photograph are usually comprised of only a few pixels. And the larger the image, the smaller those stars are likely to be. So if I were to simply use a big, panoramic photo of the sky‚Äìor even just a rendering of one‚Äìthe stars would barely be visible. And if those stars do happen to be visible, there‚Äôs a high likelihood that they‚Äôll appear as little mushy blotches on the sky. Some of this has to do with how images are compressed, but it‚Äôs also a common side-effect when working with textures that get displayed over such a large area‚Äìand there‚Äôs no way around the fact that the sky needs to be fairly large in most scenes. And the solution of using an incredibly high-resolution image will quickly fail when you realize just how large the file size would need to be.

The other issue with starry backgrounds is that you‚Äôre somewhat stuck with that particular arrangement of stars. In the previous clip, you‚Äôll notice that sky has very prominent stars‚Äìalmost to the point of looking like clouds. Maybe I just want a few stars here or there‚Äì or even cartoony five-pointed stars and planets. It would be nice to be able to separate the sky from whatever elements appear within it.

Geometry Nodes to the Rescue!

If you‚Äôre brand new to Blender‚Äôs geometry nodes system, this might seem a bit daunting. There‚Äôs another tutorial that goes over the basics (and beyond) of geometry nodes, so you may want to check that out first. But I will try to keep things simple here. Besides, this is actually a simpler use of geometry nodes and a decent place to start.

A Star is Born

We‚Äôre eventually going to need a star to scatter across our sky so I‚Äôll just build it now. All it consists of is a single mesh plane (2 triangles‚Äì no need for any subdivisions) with a very simple texture on it. I made a white circle with a slightly soft edge, but each star will be so far away, any detail is likely to be lost. You can see in the image below that I‚Äôm making sure I set the ‚ÄòBlend Mode‚Äô to either ‚ÄòAlpha Blend‚Äô or ‚ÄòAlpha Clip‚Äô (depending on how soft the edge needs to be). My image didn‚Äôt have any built-in alpha (transparency) so I had to use the ‚ÄòColor‚Äô output to feed the Alpha input of the Principled BSDF node.

I‚Äôll name my mesh ‚ÄòStar.001‚Äô so I can easily find it later.

Don‚Äôt worry about the size of the star mesh. Blender likely made it 1 or 2 meters (squared) but we will be controlling the size in a different way.

üí°
I left it out of the image above, but I also ended up connecting the star image‚Äôs ‚ÄòColor‚Äô output to the Principled BSDF node‚Äôs ‚ÄòEmission‚Äô input and turned up the ‚ÄòEmission Strength‚Äô a bit. I‚Äôm going to want the stars to look illuminated, so this will be important once it's in Hubs.

What about this Geometry Nodes graph thing?

We need an object to hold the Geometry Nodes modifier (and graph), but it can be anything you want‚Äì our graph will replace whatever you choose. I usually make a default cube or even a single vertex, it truly does not matter.

What the object needs is a Geometry Nodes modifier:

Open up a Geometry Nodes workspace (top tab menus) and now we can see what happens when we make a new graph for it. This can be done either on the Geometry Nodes modifier, or within the Geometry Nodes window at the top.

All the default graph does is take the existing geometry of whatever the graph is on (in my case, a cube) and pass that to the output‚Äì essentially doing nothing.

Not too exciting yet, is it?

We can delete the input node completely. Why? Well, we‚Äôre not interested in the cube‚Äìit‚Äôs simply a container for this geonodes modifier and the graph it holds. When we delete the input, there is no longer any geometry to pass on so the result is our cube disappearing.

Instead, we‚Äôre going to use this system of graphical nodes to generate our own geometry‚Äìin this case, a dome that will become our sky to hold the stars.

üî§
You probably will want to rename your starting object. Mine kept confusing me because it was called ‚ÄòCube‚Äô. Something like ‚ÄúStarrySky‚Äù might make more sense.

You can see toward the end of the clip above that it‚Äôs very easy to make adjustments to the icosphere using the node. This non-destructive editability is the key to what makes geometry nodes so useful. As we move on, this will become more and more apparent‚Äìand you‚Äôll understand why I didn‚Äôt just start by manually making an icosphere mesh at the very beginning.

Scattering the Stars

This next concept is covered by just about every other geometry nodes video tutorial online, but in the interest of keeping you here, I‚Äôll go over it anyway.

First, we must use a node called ‚ÄòDistribute Points on Faces‚Äô. This node will look at the incoming geometry (our newly generated icosphere) and randomly place points on the faces. For a better understanding, you can refer back to the article mentioned earlier.

Next, we'll add another node to our graph:

In simpler terms, the graph is using the ‚ÄòPlane‚Äô (single star) object and ‚Äòinstancing‚Äô it upon each new point that the ‚ÄòDistribute‚Äô node created.

The problem we‚Äôve created now is that we‚Äôve only specified which points to place them on‚Äì we didn‚Äôt say how we wanted them oriented so they default to however the original mesh was facing.

First, I need to lower the scale of the instanced star planes so I can see what‚Äôs going on. It will be easier to understand their orientation if they‚Äôre a little smaller. Here, I just type in new Scale values inside the ‚ÄòInstance on Points‚Äô node.

For the orientation, I need to get each instanced star plane to align with the dome itself‚Äì so each star plane is facing the center of the icosphere. For this, we need a new node called ‚ÄòAlign Euler to Vector‚Äô. We will be taking the ‚ÄòNormal‚Äô output of the ‚ÄòDistribute‚Äô node and plugging it into the ‚ÄòVector‚Äô input of our new node. Then, we pipe the ‚ÄòRotation‚Äô output into the ‚ÄòRotation‚Äô input of the ‚ÄòInstance on Points‚Äô node. (That's quite a mouthful...thank your stars for illustrations.)

üí°
Depending on the orientation of your starting instance mesh, you may need to try out the various Axis settings of the ‚ÄòAlign Euler To Vector‚Äô node, just like I did at the end of the clip above.

One last thing about orientation‚Ä¶

You may not see it at first, but because our starting star mesh is a plane, it only has a surface on one side (the top). However, our material setup is hiding this fact by drawing the texture on both sides. Personally, I like to know which way things are facing even when they appear to be just fine. It avoids potential problems down the line and gives you a better command of your scene.

In the star material we made earlier, I‚Äôm going to find those same settings I used to set the ‚ÄòAlpha Blend‚Äô and check that box called ‚ÄòBackface Culling`. This makes it so the material only draws on the front faces of our star mesh‚Äìand subsequently affects all the instanced stars.

Hmm‚Ä¶ so now we can see that the stars are actually facing outward instead of inward toward the scene. There are several ways to flip them around, but the easiest way would be to flip the faces of the icosphere they were scattered across. Guess what? There‚Äôs a node for that: ‚ÄòFlip Faces‚Äô.

üòñ
If you‚Äôre starting to feel a little overwhelmed by the sheer number of geometry nodes, don‚Äôt worry. The best way to learn them is one at a time‚Äì using them as you need them. Before you know it, they will become part of your 3d vocabulary.

Scaling the stars (a better way this time)

Our starry sky is starting to shape up, but it still looks more like a big disco ball than a planetarium. Obviously, the stars are still too big. Worse though, is that they‚Äôre all exactly the same.

Fortunately, Blender‚Äôs geometry nodes have lots of ways to introduce randomness, which is exactly what we need for our stars to look more natural.

Here, I‚Äôll use a handy technique for making new nodes‚Äì pay attention to how I do it:

Once that ‚ÄòRandom Value‚Äô is connected to the ‚ÄòScale‚Äô of ‚ÄòInstance on Points‚Äô, you‚Äôll likely see your stars go crazy. They‚Äôre just showing the default values‚Äìin this case, a minimum of zero and maximum of 1 on all three axes. Try changing those values so they make more sense‚Ä¶

It‚Äôs not easy, right? First of all, you‚Äôre trying to change 6 values (well, 4 that actually do something). Meanwhile, the starting size of the original star mesh is probably still too big. It‚Äôs a bit of a headache trying to figure out the end result.

Before you panic, notice that the ‚ÄòRandom Value‚Äô node has a dropdown menu. You can change it to read ‚ÄòFloat‚Äô instead of ‚ÄòVector‚Äô. This will simplify the node so that it only outputs 2 random values (min/max). The side effect (and one we actually want), is that each star will always be scaled uniformly on all axes. Otherwise, each star would possibly be scaled differently on X & Y‚Äìa potentially cool look, but not what we‚Äôre going for here.

üîå
You might find that changing a node‚Äôs settings will sometimes disconnect its wires. Blender usually does this when the result would be invalid and to avoid compilation errors.

We can add another node to control the overall scale. This kind of operation is quite common in geometry nodes so it‚Äôs a good one to remember:

We can multiply the scale values coming from the ‚ÄòRandom Value‚Äô node with a number that will reduce them all. For this, we can use a ‚ÄòMath‚Äô node and set it to ‚ÄòMultiply‚Äô.

By adjusting the random min/max and overall scale, we can now achieve a better result.

Randomizing star types

If you‚Äôre finding that all the stars look too similar, even though they have different scales, it‚Äôs pretty easy to get your graph to use different stars‚Äìyou just have to make them first.

In cooking show fashion, I‚Äôve gone ahead and pre-made a variety of stars. These are all just duplicates of the original star mesh but I‚Äôve given them new materials with different textures.

üß†
A smarter way of handling multiple textures would be to put all the different images of stars onto one texture sheet or ‚Äòatlas‚Äô. Then, you could just offset their UVs to make them look different. This would have the advantage of using less texture memory and cause fewer draw calls to the GPU‚Äìin Hubs, or any real-time rendering engine.

The key here is making sure you put all these star meshes into their own Collection. (Note the circled collection in the image above.)

Just like before when we dragged and dropped the star mesh into our graph, we‚Äôll do the same thing‚Äìonly this time, we‚Äôll drag the entire collection. The graph creates a ‚ÄòCollection Info‚Äô node.

Now, we need to replace the original star mesh node with the collection node. You‚Äôll notice that suddenly each and every star has been replaced‚Äì with ALL the stars. We don‚Äôt want to use the entire collection each time, so we need to turn on ‚ÄòSeparate Children‚Äô on the ‚ÄòCollection Info‚Äô node. Last, we need to turn on the checkbox called ‚ÄòPick Instance‚Äô on the ‚ÄòInstance on Points‚Äô node.

It wasn‚Äôt shown in the clip above, but now you can see each star getting a different mesh from the collection.

Wrapping up (our UVs)

To finalize everything we‚Äôve done with the geometry nodes graph, we‚Äôll need to do a couple things. First, our graph needs a specialized node called ‚ÄòRealize Instances‚Äô at the end of it. This will take each star mesh instance and convert it into a regular mesh. However, (for reasons I‚Äôd rather not go deeply into here), adding that node will have the result of removing the UV coordinates from the stars, thus making them have no textures.

To avoid this problem, there‚Äôs a somewhat quirky workaround. We must first Apply the Geometry Nodes modifier. This will actually produce a new set of UVs, but they show in a different place from regular UVs you might be familiar with. There is a section in the Object Data Properties tab called ‚ÄòAttributes‚Äô. After doing the Apply of the modifier, you‚Äôll now have a couple sets of data inside the ‚ÄòAttributes‚Äô. Select the one called ‚ÄòUVMap‚Äô, then use the down arrow icon and choose ‚ÄòConvert Attribute‚Äô. In the subsequent pop-up, choose 'UV Map', and hit Ok.

As soon as you convert the UVMap data, the textures reappear on the model. Whew!

ü§î
You might be thinking, ‚ÄúBut I didn‚Äôt necessarily want to collapse everything down and lose my graph.‚Äù But remember, that graph still exists and can simply be applied to a new object to make another starry sky. The only thing you want to keep around are your star meshes. Anytime you use that graph, it needs to reference those stars again.

Putting It All Together

We now have this nice single mesh that‚Äôs comprised of many small meshes, each with random star textures. If you didn‚Äôt make the original sphere quite large enough for your scene, you could always go back a few steps and increase the size of the icosphere in the graph.

But you may also find it simpler to go into ‚ÄòEdit‚Äô mode, set your 3d cursor to the origin, then scale all the stars together. This will also have the side-effect of making the stars themselves bigger. But you can switch your ‚ÄòTransform Pivot Point‚Äô dropdown to ‚ÄòIndividual Origins‚Äô, then scale all the star faces individually at the same time.

Finally, we can use this ball of stars on its own (plain background), or as a layer on top of whatever sky background you choose.

Taking it further...

The techniques you‚Äôve seen here could be applied to all kinds of things, from clouds to background trees and buildings. Instead of scattering things across a sphere, you can experiment using simpler meshes, or using the ‚ÄòInstance on Points‚Äô node to simply put a model at each vertex (point) of another model.

In another scene, I used this same technique to make stars, but then I added the ‚Äòuv-scroll‚Äô component (as seen in this tutorial series) to make some of the stars appear to ‚Äòtwinkle‚Äô slightly if you happen to look at them long enough to notice.

If this was your first foray into using Geometry Nodes, I think congratulations are in order. The concepts aren‚Äôt always easy to grasp right away, but the more you use them, the easier it becomes. I‚Äôm still learning them myself, so come drop by the Hubs Discord and maybe you could share your tricks with the community as your skills improve.
